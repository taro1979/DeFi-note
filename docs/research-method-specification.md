## 2. 背景と目的

DeFi History Atlas は、DeFiプロトコルの進化、主要カテゴリ、失敗事例、攻撃事例、地域別の用語差を横断的に整理する調査型コンテンツである。DeFi領域では、公式ドキュメント、ホワイトペーパー、監査レポート、ガバナンス投稿、オンチェーンデータ、攻撃データベース、SNS投稿などが分散しており、特に短命プロトコルやRug済みプロトコルでは公式サイトが消失している場合がある。

本仕様書は、`research-method.html` に掲載する調査方法の内容を、業務で再利用可能な仕様として整理することを目的とする。調査担当者が同一基準で情報を収集、検証、記録できる状態を作り、コンテンツの信頼性と再現性を担保する。

| 対象情報源 | 公式Docs、Whitepaper、GitBook、Governance Forum、GitHub、監査レポート、攻撃DB、TVLサイト、アダプター実装、Wayback Machine、Explorer、X投稿、ニュース記事、多言語圏資料 |

## 4. 基本方針

1. 英語一次資料を事実確認の基準とする。
2. 公式資料だけでなく、消失したプロトコルの痕跡、攻撃事例、オンチェーン実装、古いガバナンス投稿を横断的に確認する。
3. 多言語圏で異なる用語が使われるため、日本語、中文、英語、韓国語、ベトナム語の検索語を併用する。
4. 成功したプロトコルだけでなく、短命、攻撃済み、Rug済み、UI消失済みのプロトコルも調査対象に含める。
5. 日付、ステータス、TVL、カテゴリ分類は市場環境やドキュメント更新により変化するため、記録時点と参照元を明示する。

## 5. 業務要件

### 5.1 調査対象の網羅性

調査担当者は、現在も稼働している主要プロトコルだけでなく、過去に重要な影響を与えたプロトコル、失敗したプロトコル、攻撃を受けたプロトコル、短期的に流行したカテゴリを含めて調査すること。

対象例は次の通り。

| 分類 | プロトコル例 |
| --- | --- |
| 主要プロトコル | Uniswap、Curve、Aave、Maker、Frax、Pendle、Ethena、Morpho |
| 攻撃・失敗事例 | Harvest、Cream、PancakeBunny、Euler、Kyber、Curve/Vyper |
| トレンドカテゴリ | LSTfi、RWA、Restaking、Basis Trading、Food Farm、OHM Fork、Real Yield |

### 5.2 情報源の階層

情報源は信頼度に応じて分類し、事実確定時には高信頼度の情報源を優先する。

| 優先度 | 情報源 | 用途 |
| --- | --- | --- |
| A | 公式Docs、Whitepaper、公式GitBook、Governance Forum、GitHub | 設計意図、仕様、用語、実装、意思決定の確認 |
| A | 監査レポート、Post Mortem、公式インシデント報告 | 攻撃原因、影響範囲、修正内容の確認 |
| B | DeFiLlama、Token Terminal、Dune Dashboards、Explorer | TVL、収益、利用状況、オンチェーン実態の確認 |
| B | Rekt、Immunefi、監査会社ブログ | 攻撃済み、短命、Rug関連プロトコルの発見 |
| C | ニュース記事、ブログ、X投稿、当時のSNS投稿 | 当時の市場熱量、文脈、コミュニティ認識の補足 |
| C | 多言語解説記事、地域メディア | 地域別の用語差、注目テーマ、検索補助 |

## 6. 多言語調査仕様

### 6.1 多言語調査の目的

DeFiに関する一次資料は英語が中心である一方、地域別の解説、検索語、ユーザー理解は言語圏ごとに異なる。多言語調査では、同一概念に対する言語別の表現差を把握し、英語資料だけでは拾えない地域別の注目テーマや過去事例を補完する。

### 6.2 言語別キーワード

| 言語 | 使用語例 | 確認方針 |
| --- | --- | --- |
| 日本語 | 流動性マイニング、自動マーケットメーカー、リキッドステーキング | 日本語公式ドキュメントが存在する場合は、公式翻訳を確認する。例: Rocket Pool 日本語ドキュメント |
| 中文 | 自动做市商、收益耕作、流动性挖矿、流动性质押 | DODO PMM算法、Rocket Pool中文など、公式中文資料を優先して確認する。 |
| English | docs、whitepaper、audit、academic paper、governance forum | 事実関係を確定するための基準資料として扱う。 |
| 한국어 | 디파이、유동성 채굴、리퀴드 스테이킹 | Rocket Pool 한국어など、公式翻訳が存在する領域では確認対象に含める。 |
| Tiếng Việt | tài chính phi tập trung、pool thanh khoản、farming、liquid staking | Coin98、Upsideなどの地域向け解説を補助資料として確認する。 |

### 6.3 多言語資料の利用ルール

1. 多言語資料は、原則として検索補助、用語差の確認、地域別文脈の補完に利用する。
2. 事実関係、仕様、数値、インシデント原因は、英語一次資料または公式資料で再確認する。
3. 翻訳語を採用する場合は、同一ページ内で表記ゆれが発生しないように用語集を整備する。
4. 非公式解説記事を引用する場合は、一次資料に到達できるか確認する。

## 7. 調査プロセス仕様

### 7.1 公式DocsとWhitepaperの確認

現在も資料が残るプロトコルについては、公式Docs、GitBook、Whitepaper、Governance Forumから設計意図を確認する。特に、Uniswap、Curve、Aave、Maker、Frax、Pendle、Ethena、Morphoなどは公式資料の品質が高く、この方法を主軸とする。

確認項目は次の通り。

| 項目 | 確認内容 |
| --- | --- |
| プロトコル概要 | 提供機能、対象ユーザー、対応チェーン、主要ユースケース |
| 技術設計 | AMM設計、担保設計、清算設計、オラクル、手数料、Vault、Gauge、Restakingなど |
| 経済設計 | トークン排出、インセンティブ、収益分配、手数料、ガバナンス権限 |
| リスク | スマートコントラクトリスク、清算リスク、オラクルリスク、管理者権限、スラッシングリスク |
| 変更履歴 | バージョン、仕様変更、ガバナンス提案、移行計画 |

### 7.2 攻撃DBからの逆引き

攻撃DB、インシデント記事、監査会社ブログを利用し、短命または攻撃済みプロトコルを抽出する。成功史だけでは見えないカテゴリ、実装ミス、資金流出パターンを把握することを目的とする。

参照先の例は次の通り。

| 情報源 | 用途 |
| --- | --- |
| Rekt Leaderboard | 大規模攻撃、Rug、損失規模、時系列の確認 |
| Immunefi Research | 損失レポート、攻撃傾向、カテゴリ別被害の確認 |
| 監査会社ブログ | 脆弱性分析、技術的原因、再発防止策の確認 |
| Post Mortem | 当事者による原因説明、対応履歴、補償方針の確認 |

### 7.3 DeFiLlamaとアダプター実装の観察

DeFiLlamaなどのTVLサイトやアダプター実装は、UIや公式サイトが消えた小規模プロトコルの名前を確認するために利用する。カテゴリの変遷を見ることで、LSTfi、RWA、Restaking、Basis Tradingなどの市場トレンドを把握する。

確認項目は次の通り。

| 項目 | 確認内容 |
| --- | --- |
| TVL推移 | 急増、急減、消失、カテゴリ移動 |
| カテゴリ | Lending、DEX、Yield、Liquid Staking、RWA、Restakingなど |
| アダプター | コントラクトアドレス、対象チェーン、算出ロジック |
| 関連プロトコル | Fork、統合先、後継プロトコル、エコシステム内の位置付け |

### 7.4 古いニュースとブログの確認

DeFi Summer、Food Farm、OHM Fork、Terra、FTX後のReal Yieldなどは、当時の記事やSNS上の熱量を確認しないと文脈を把握しにくい。ニュースやブログは市場認識の補足として利用し、正確性は公式Docsまたは一次資料で再確認する。

### 7.5 多言語キーワード検索

中文の「流动性挖矿」「自动做市商」、韓国語の「유동성 채굴」、ベトナム語の「pool thanh khoản」などで検索し、地域別に注目されたプロトコルや説明の違いを確認する。

検索時には、英語キーワード、チェーン名、カテゴリ名、攻撃関連語を組み合わせる。

### 7.6 コードとガバナンスの照合

深掘り調査では、GitHub、監査レポート、Snapshot、Governance Forum、Token Terminal、Dune Dashboardsを利用し、設計、利用状況、収益、意思決定を照合する。

照合観点は次の通り。

| 観点 | 確認内容 |
| --- | --- |
| コード | コントラクト構造、権限、Proxy、Timelock、Vault、Strategy、MasterChef |
| 監査 | 監査範囲、重大指摘、修正状況、監査対象外領域 |
| ガバナンス | 提案内容、投票結果、実行日、利害関係者の議論 |
| 収益 | 手数料、プロトコル収益、ユーザー向け利回り、補助金依存 |
| 利用状況 | TVL、取引量、ユーザー数、アクティブな市場またはVault |

### 7.7 Xの古い投稿の確認

Xでは、公式名だけでなく、Ticker、Chain、Rug、Exploit、Post Mortem、Thread、Ape、Farmなどの語を組み合わせて検索する。PancakeBunnyのように時期が明確な事例では、検索期間を指定して当時の投稿を確認する。

例:

```text
"PancakeBunny" exploit until:2021-06-01 since:2021-05-01
```

### 7.8 Wayback Machineによる消失Docsの確認

Rug済みFarmや短命プロトコルでは公式サイトが消失していることが多い。Wayback Machineを利用し、当時のドメイン、Docs、GitBook、Medium、監査文言、APY表示、権限説明を確認する。

確認すべき内容は次の通り。

| 項目 | 確認内容 |
| --- | --- |
| 当時の訴求 | APY、Buyback、Burn、Deposit Fee、Referral |
| 権限説明 | Owner権限、Upgrade権限、Timelock、ProxyAdmin |
| 監査表示 | 監査会社名、監査範囲、監査日、未監査の明記 |
| ドキュメント構造 | Whitepaper、Docs、GitBook、Medium、FAQ |

### 7.9 Explorerからの名前抽出

BscScan、Polygonscan、SnowtraceなどのExplorerを利用し、MasterChef、Strategy、Vault、Timelock、ProxyAdmin、Owner Transactionを追跡する。UIが消えたプロトコルでも、コントラクト名、イベント名、トランザクション履歴は残るため、消失プロトコルの復元に利用する。

## 8. 検索クエリ仕様

調査担当者は、次の検索クエリを起点として、対象プロトコル名、チェーン名、カテゴリ名、時期を追加して検索する。

```text
"MasterChef" "deposit fee" BSC farm rug
"upgradeTo" "Vault" "BSC" rug
"OHM fork" "buyback" rekt
"Rug" "BSC" "ファーム"
"流动性挖矿" "跑路" DeFi
"유동성 채굴" "러그풀"
"pool thanh khoản" "rug pull"
```

検索結果を採用する場合は、参照URL、取得日、参照した主張、一次資料への到達可否を記録する。

## 9. コンテンツ要件

### 9.1 ページ構成

`research-method.html` は、次の構成を満たすこと。

| セクション | 内容 |
| --- | --- |
| 多言語調査メモ | 言語別のDeFi用語、参照先、利用方針を掲載する。 |
| 調査の掘り方 | DeFi史を深く調査するための情報源、調査手順、検索手法を掲載する。 |
| 検索クエリ例 | 具体的な検索語を掲載し、調査担当者が再利用できる状態にする。 |

### 9.2 表現方針

1. 読者が実務でそのまま使える具体性を持たせる。
2. 情報源の名称、プロトコル名、検索語は省略せずに記載する。
3. 断定が難しい内容は、根拠、時期、参照元を明示する。
4. 攻撃、Rug、失敗事例を扱う場合は、事実と評価を分離して記述する。
5. 多言語語彙は原語表記を維持し、必要に応じて日本語の説明を添える。

## 10. 品質基準

| 基準 | 内容 |
| --- | --- |
| 正確性 | 主要な事実は公式資料、一次資料、または信頼できる分析資料で確認されていること。 |
| 再現性 | 他の担当者が同じ検索語と情報源から調査を再現できること。 |
| 網羅性 | 成功事例、失敗事例、攻撃事例、消失プロトコル、多言語資料を横断していること。 |
| 透明性 | 参照元、確認日、推測と事実の区別が明確であること。 |
| 更新容易性 | 市場やドキュメント更新に応じて、対象ページと仕様書を継続更新できること。 |

## 12. 受け入れ基準

本仕様に基づくページまたは調査成果物は、次の条件を満たす場合に受け入れ可能とする。

1. 多言語調査メモに、日本語、中文、英語、韓国語、ベトナム語の用語例と参照方針が含まれている。
2. 調査方法に、公式Docs、攻撃DB、DeFiLlama、ニュース、ブログ、多言語検索、コード、ガバナンス、X、Wayback、Explorerの観点が含まれている。
3. 検索クエリ例が具体的で、調査担当者が再利用できる。
4. 情報源の信頼度に応じた確認方針が明示されている。
5. 攻撃済みまたは消失済みプロトコルの調査方法が含まれている。
6. 事実確認の基準として英語一次資料を優先する方針が明示されている。
7. 市場情報やプロトコルステータスが変化しうることを前提に、更新可能な文書構造になっている。

## 13. 運用ルール

1. `research-method.html` の内容を変更した場合は、本仕様書も合わせて更新する。
2. 新しい調査対象カテゴリを追加した場合は、検索語、情報源、確認観点を追記する。
3. リンク切れ、Docs消失、プロトコル停止を確認した場合は、Wayback Machine、Explorer、GitHub、Duneなどの代替情報源を確認する。
4. 参照元が二次資料のみの場合は、該当箇所を未確定情報として扱い、一次資料確認の課題として残す。
5. 重要な数値、攻撃額、日時、バージョン、ガバナンス結果は、複数情報源で照合する。
