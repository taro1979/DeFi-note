# check-uiux — UIUX 軽量サニティチェック

UI を修正するたびに自動発火するサニティチェック。速度よりも精度を優先する — 全項目を丁寧に確認し、見落としゼロを目指すこと。時間制約はない。

## 実行手順

1. 直前に変更されたファイルを特定（git diff --name-only 等、または会話コンテキストから判断）
2. 変更ファイルに対して以下の項目を **Grep のみ** で高速チェック
3. 結果を1テーブルで出力

## チェック項目

### 1. カラーハードコード
変更ファイル内に `#[0-9a-fA-F]{3,8}` や `rgb(` がないか Grep。
CSS変数やTailwindクラスでなくインライン hex/rgb があれば ⚠️。
（SVG内の外部ブランドカラーや意図的なものは許容）

### 2. 角丸チェック
変更ファイル内の `rounded-` クラスを Grep。
`rounded-sm` や `rounded-md` があれば ⚠️（`rounded-lg` 以上を推奨）。
`rounded-xl`, `rounded-2xl`, `rounded-full` は ✅。

### 3. スペーシング
**LP（LandingPage.tsx、landing/ 配下）の場合**:
- `max-w-7xl` 以外のコンテナ幅があれば ⚠️
- セクション要素の `py-` が `py-20` 未満なら ⚠️

**アプリ画面（pages/ 配下の一般ページ）の場合**:
- `container` / `max-w-` でコンテンツ幅が制御されているか確認（なければ ⚠️）
- `p-0` や `m-0` でパディングが完全に消えていないか確認

### 4. レスポンシブプレフィックス
変更ファイルに `className=` があるのに `sm:` `md:` `lg:` が一切なければ ⚠️。
（ユーティリティコンポーネント等、不要なケースは許容）

### 5. レスポンシブ詳細
- `hidden` / `block` と `sm:block` / `md:hidden` 等のペアで表示切替しているか
- `grid-cols-` に `md:grid-cols-` 等のレスポンシブ指定があるか
- `flex-col` → `md:flex-row` 等、モバイルで縦並び・デスクトップで横並びのパターンがあるか
- `w-full` / `w-[固定値]` がレスポンシブ切替なしに混在していないか
- `text-` サイズに `md:text-` 等のレスポンシブ指定があるか（見出し系は必須）

### 6. a11y 基本
変更ファイル内の `<button` に `aria-label` や表示テキストがあるか。
`<img` に `alt` があるか。
アイコンのみのボタンに `aria-label` がなければ ❌。

### 7. フォーカスリング
変更ファイル内の `<button` `<a` `<input` に `focus-visible:` や `focus:ring` があるか。
インタラクティブ要素にフォーカススタイルが一切なければ ⚠️。

### 8. ローディング状態
`useQuery` / `useMutation` / `fetch` / `tRPC` の近くに `loading` / `isLoading` / `isPending` / `Skeleton` / `Spinner` の分岐があるか。
非同期データ取得があるのにローディング表示がなければ ⚠️。

### 9. エラーUI（必須）
`useQuery` / `trpc.*.useQuery` を使うコンポーネントに `isError` / `error` の分岐と、ユーザー向けのエラー表示（エラーメッセージ、リトライボタン等）があるか。
**判定ロジック**: 変更ファイル内で `useQuery` を Grep → 同じコンポーネント（関数スコープ）内に `isError` または `error` を使った条件分岐（`if (isError)` / `isError &&` / `error ?` 等）があるか確認。
- エラー分岐がなければ ❌（必須違反）。
- エラー分岐はあるが `retry` / `refetch` ボタンがなければ ⚠️（推奨）。
- タイムアウトや通信エラーに対する表示が考慮されていれば ✅。

### 10. 空状態
リスト表示で `length === 0` / `empty` / `no data` の空状態UIがあるか。
データ取得結果が空配列になりうる箇所に空状態表示がなければ ⚠️。

### 11. i18n ハードコード（必須）
UIに表示される日本語テキストが `t()` を経由せず直書きされていないか。

**判定ロジック**:
1. 変更ファイル内で日本語文字（ひらがな・カタカナ・漢字）を含む文字列リテラルを Grep
2. 以下の **JSX 属性・要素内** に日本語リテラルがあれば ❌（必須違反）:
   - JSX テキストノード: `<p>日本語</p>`, `<span>日本語</span>` 等
   - `placeholder="日本語"`, `label="日本語"`, `title="日本語"`, `alt="日本語"`
   - `toast("日本語")`, `toast.success("日本語")` 等のユーザー向けメッセージ
   - テンプレートリテラル内: `` `${name}様` `` 等
3. 以下は **許容**（偽陽性として除外）:
   - コメント (`//`, `/* */`)
   - `console.log` / `console.warn` / `console.error` 内
   - `import` 文
   - 型定義・interface 内のコメント
   - i18n 非対応と明記されたページ（管理画面等）
   - テストファイル (`*.test.tsx`)
   - `aria-label` に日本語がある場合（`t()` 推奨だが既存は許容）

**重大度**:
- JSX テキスト・属性に日本語リテラル → ❌（必須違反、`t()` に置き換え必要）
- toast メッセージに日本語リテラル → ❌
- 上記以外（変数名等）→ 許容

### 12. トースト統一
`toast(` の呼び出しが `sonner` 経由か。
独自の `alert()` や `window.confirm()` があれば ❌。

### 13. z-index 管理
`z-[数値]` のハードコード Grep。Tailwind の `z-10` `z-50` 等の定義済みクラスは ✅。
`z-[999]` `z-[9999]` 等の任意値があれば ⚠️。

### 14. テキスト切れ
長文を表示する要素（テーブルセル、カード内テキスト、ラベル）に `truncate` / `line-clamp` / `overflow-hidden` / `text-ellipsis` があるか。
多言語で文字数が変わるため、明らかに長文が入る箇所に切れ対策がなければ ⚠️。

### 15. 右下ボタン配置禁止
変更ファイル内に `sticky bottom` / `fixed bottom` と `right` / `justify-end` の組合せで、ボタンが画面右下に固定されるパターンがないか Grep。
本番環境にプラットフォームのフローティングボタンが常駐する場合、右下固定のアクションボタンは干渉する。
右下固定配置があれば ❌。

### 16. 未保存離脱防止
フォームページ（`<form` / `handleSubmit` / `useMutation` を含むページ）で、未保存変更時のページ離脱防止が実装されているか。
以下のいずれかが存在するか確認:
- `beforeunload` イベントリスナー
- `isDirty` / `formTouched` / `dirty` 等の変更追跡ステート
- `AlertDialog` / `confirm` 等の離脱確認UI
フォームページなのに離脱防止が一切なければ ⚠️。
（ダイアログ内の小さなフォームや検索フォーム等、軽量なものは許容）

### 結果の読み方と行動

**❌（必須違反）が1つでもある場合**:
- 完了報告せず、その場で修正してからチェックを再実行する
- `/dev-impl` や `/self-review` の途中で発火した場合も同様 — 修正が終わるまで次のフェーズに進まない

**⚠️（推奨）のみの場合**:
- 該当行と修正提案を1〜2行で添えて報告する
- 修正するかはユーザー判断に委ねてよい

**N/A（該当なし）の扱い**:
- 該当する要素がファイル内に存在しない項目は `—`（該当なし）として ✅ 扱いにする
- 例: `useQuery` を使わないコンポーネントでの項目9、フォームでないページでの項目16

詳細なレビューが必要な場合は `/review-lp-uiux` を案内する。
