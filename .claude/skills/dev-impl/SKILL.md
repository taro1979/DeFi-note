# 仕様駆動実装ワークフロー

このワークフローの目的は、確定済みの仕様書に基づいて、TDD 原則で実装と検証を完了させることです。

## フェーズ 0: セッション名の設定

タスクの内容を反映したセッション名を設定する。`/rename {タスク概要}` を実行する。

- 例: `/rename stripe-webhook-fix`、`/rename add-export-csv`
- 英語のケバブケースで、短く具体的に（2〜4語）
- 既にセッション名が設定済みで、同じタスクの続きなら変更不要

## フェーズ 1: 事前調査

1. **仕様書の有無を確認**: `spec/feature-*.md` を探し、対象タスクに対応する仕様書があるか確認
2. **フォールバック判定**: 仕様書が見つからない、または今回の依頼を十分にカバーしていない場合は、このスキルで実装を進めず `dev-spec` を先に実行する
3. **仕様書を熟読**: Acceptance Criteria を把握
4. **アーキテクチャを確認**: `docs/architecture.md` で変更レイヤーと依存方向を確認
5. **既存実装を調査**: 対象ファイルを Read / Grep で確認し、影響範囲を把握
6. **実装計画を宣言**: どのファイルに何を変更するかを短く共有

## フェーズ 2: テスト駆動開発

### Red
- Acceptance Criteria をテストケースに変換する
- テストが失敗することを確認する

### Green
- 最小限の実装でテストを通す
- 依存方向、ログ規約、例外処理規約、既存パターンを守る

### Refactor
- 可読性、重複、責務分離を改善する

## フェーズ 3: 検証

以下を基本として、存在するコマンドを実行する。

```bash
pnpm test
pnpm check
pnpm build
```

- コマンドが存在しない場合は、等価コマンドで代替する
- テスト、型チェック、ビルドのうち何を実行したかを完了報告に含める
- Self-Review を実行する（Skill ツールで `self-review` を呼び出す）

## フェーズ 4: TODO更新と完了報告

### TODO更新
検証がパスしたら、`todo.md` を確認し:
- 実装した機能に対応する `[ ]` 項目を `[x]` に更新
- 該当項目がなければ、適切なセクションに完了項目として追記

### 完了報告
- 変更ファイル一覧と差分サマリー
- テスト結果
- Acceptance Criteria の充足状況
- 実行した検証コマンド
