# Test Coverage Audit

テストカバレッジのギャップを特定し、追加すべきテストの優先度をレポートする。

## ワークフロー

### Phase 1: ユニットテストカバレッジ収集

プロジェクトのテストランナーでカバレッジを取得する。

```bash
# vitest の場合
npx vitest run --coverage 2>&1; echo "EXIT_CODE=$?"
# jest の場合
# npx jest --coverage 2>&1; echo "EXIT_CODE=$?"
```

**注意**: 終了コードが 0 以外の場合、テスト自体が失敗している。カバレッジ分析に進む前にテスト失敗を報告し、修正を優先するよう提案する。

istanbul テキストレポートからファイルごとの Stmts / Branch / Funcs / Lines を取得する。

### Phase 2: テストファイル対応マッピング

1. バックエンドのソースファイル一覧を Glob で取得（`server/**/*.ts` or `src/**/*.ts`、除外: `*.test.ts`, `*.spec.ts`）
2. 対応する `.test.ts` が存在するか確認
3. テストファイルが存在しないソースファイルをリストアップ

### Phase 3: E2E テストカバレッジ確認

1. `e2e/*.spec.ts` または `tests/**/*.spec.ts` の一覧を取得
2. ページコンポーネントの一覧を取得（`src/pages/*.tsx` 等）
3. 各 E2E spec ファイルの中身を Grep で確認し、実際にどのページ/ルートをテストしているかを特定する（ファイル名の一致だけに頼らない）
4. E2E テストでカバーされていないページ・機能をリストアップ
5. 各 E2E テストファイルの `test()` / `test.describe()` ブロック数を集計

### Phase 4: ギャップ分析

以下の観点で未カバー領域を分類:

| カテゴリ | 判定基準 |
|---|---|
| **Unit 未テスト** | 対応 `.test.ts` が存在しないソースファイル |
| **Unit 低カバレッジ** | カバレッジレポートで Lines < 50% のファイル |
| **E2E 未テスト** | 対応 `.spec.ts` がない主要ページ |
| **E2E 浅いテスト** | テストケース数が極端に少ない（< 3）spec |
| **Branch 未カバー** | Branch カバレッジが特に低い（< 30%）ファイル |

### Phase 5: 優先度レポート出力

以下のフォーマットでレポートを出力:

```
## テストカバレッジ監査レポート

### サマリー
- ユニットテスト: XX ファイル中 YY ファイルにテストあり (ZZ%)
- E2E テスト: XX ページ中 YY ページにテストあり (ZZ%)
- 全体 Lines カバレッジ: XX%

### 🔴 High Priority（テスト追加を強く推奨）
- [ファイル名] — 理由

### 🟡 Medium Priority（テスト追加を推奨）
- [ファイル名] — 理由

### 🟢 Low Priority（現状で許容）
- [ファイル名] — 理由

### 推奨アクション
1. ...
2. ...
```

## 優先度の判定基準

| 優先度 | 条件 |
|---|---|
| 🔴 High | ビジネスロジック（決済、認証、データ永続化）かつテストなし/低カバレッジ |
| 🟡 Medium | CRUD 操作、API エンドポイント、ユーティリティでテスト不足 |
| 🟢 Low | 設定ファイル、型定義、開発用ヘルパーなどテスト不要なもの |

## 注意事項

- フロントエンドにユニットテストがない場合は、E2E でカバーされているかを確認する
- インフラ層は統合テストでカバーされていれば許容
- カバレッジ数値だけでなく、ビジネスインパクトを考慮して優先度を判定する
