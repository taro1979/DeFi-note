# CLAUDE.md 品質チェッカー

Anthropic 公式ドキュメント・コミュニティベストプラクティス・学術研究に基づく22項目チェックリストで CLAUDE.md の品質を診断する。

### 情報ソース

- [Anthropic 公式: Memory & CLAUDE.md](https://docs.anthropic.com/en/docs/claude-code/memory)
- [Anthropic 公式: Hooks](https://docs.anthropic.com/en/docs/claude-code/hooks)
- [GitHub Blog: 2,500+ リポジトリ分析](https://github.blog/) — CLAUDE.md パターン調査
- [ETH Zurich 研究] — 非推論的情報の価値が最も高い
- [affaan-m/everything-claude-code](https://github.com/affaan-m/everything-claude-code) — エコシステム活用パターン

## ワークフロー

### Phase 0: 対象ファイル特定

1. `$ARGUMENTS` にパスがあればそのファイルを読む
2. なければプロジェクトルートの `CLAUDE.md` → `.claude/CLAUDE.md` → `AGENTS.md` → `agent.md` の順で検索
3. 見つからなければ「ファイルが存在しません。`/claude-md-gen` で生成してください」と報告して終了

### Phase 1: ドキュメント品質チェック（6項目）

対象ファイルを Read で読み込み、以下を判定する:

| # | チェック項目 | 合格基準 | 重み |
|---|---|---|---|
| D1 | CLAUDE.md が存在する | ファイルが存在し、空でない | 必須 |
| D2 | 技術スタックが記載されている | 言語・フレームワーク・主要ライブラリが **バージョン付き** で明記（例: "React 18 + TypeScript 5.4"）。バージョンなしでも名前があれば半分合格（⚠️） | 必須 |
| D3 | 禁止事項セクションがある | 「NEVER」「禁止」「絶対やらない」「MUST NOT」等のセクションが存在し、3項目以上 | 必須 |
| D4 | 実行可能コマンドが記載されている | test / build / lint / format のうち2つ以上のコマンドが明記されている | 必須 |
| D5 | コード実例が2個以上 | コードブロック（``` ）が2つ以上あり、プロジェクト固有のパターンを示している | 推奨 |
| D6 | 行数が200行以下 | root の CLAUDE.md が200行以下。超える場合は `@import` や外部ドキュメント参照で分割しているか | 推奨 |

> **根拠（D4）**: 公式ドキュメントが「ビルド・テスト・リントコマンドを最初に書け」と推奨。エージェントが最も頻繁に参照するセクション。
> **根拠（D6）**: 公式が200行以下を推奨。フロンティアLLMは150-200命令で安定し、それ以上は遵守率が低下する。

### Phase 2: リポジトリ構造チェック（5項目）

Glob / Grep / Read でリポジトリを調査し、以下を判定する:

| # | チェック項目 | 合格基準 | 重み |
|---|---|---|---|
| S1 | 機能単位のディレクトリ分割 | `features/`, `modules/`, または Vertical Slice 構造がある | 推奨 |
| S2 | 型定義にコメントがある | 主要な型定義ファイルに JSDoc / コメントが付いている（3ファイル以上サンプリング） | 推奨 |
| S3 | 構造化エラーハンドリング | エラーコード + メッセージの構造化パターンが存在する | 推奨 |
| S4 | ディレクトリ README | 主要ディレクトリに README.md が存在する（2つ以上） | 推奨 |
| S5 | validate スクリプト | `package.json` の scripts に `validate` 相当（型チェック+リント+テスト一括）がある | 推奨 |

### Phase 3: 検証基盤チェック（5項目）

| # | チェック項目 | 合格基準 | 重み |
|---|---|---|---|
| V1 | テストが存在する | テストファイルが5つ以上存在する | 必須 |
| V2 | Pre-commit フック | `.husky/`, `.git/hooks/`, `lefthook.yml`, `lint-staged` 等が設定されている | 推奨 |
| V3 | CI 設定がある | `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile` 等が存在する | 推奨 |
| V4 | カバレッジ設定 | `jest.config`, `vitest.config`, `nyc`, `c8` 等にカバレッジ設定がある | 推奨 |
| V5 | PR テンプレート | `.github/pull_request_template.md` または `.github/PULL_REQUEST_TEMPLATE/` が存在する | 推奨 |

### Phase 4: エコシステムチェック（4項目）

Claude Code の `.claude/` ディレクトリ活用度を判定する:

| # | チェック項目 | 合格基準 | 重み |
|---|---|---|---|
| E1 | ルールファイルが存在する | `.claude/rules/` 配下に `.md` ファイルが1つ以上。大規模プロジェクトでは3つ以上が理想 | 推奨 |
| E2 | Hooks が設定されている | `.claude/settings.json` または `.claude/settings.local.json` に `hooks` キーが存在し、少なくとも1つのフックが定義されている | 推奨 |
| E3 | Skills が存在する | `.claude/skills/` または `.claude/commands/` 配下にスキル定義（`SKILL.md`）が1つ以上 | 推奨 |
| E4 | プログレッシブ・ディスクロージャー | CLAUDE.md 内に `@path` インポート、外部ドキュメント参照（`docs/` へのリンク）、またはサブディレクトリ CLAUDE.md が存在する | 推奨 |

> **根拠（E1）**: 公式推奨 — ルールは CLAUDE.md と分離してトピック別に管理すると遵守率が向上。
> **根拠（E2）**: CLAUDE.md は確率的（約80%遵守）だが、Hooks は100%確定的。重要なルールは Hooks で強制すべき。
> **根拠（E3）**: スキルはドメイン知識とワークフローをオンデマンドでロードし、コンテキストウィンドウを節約する。
> **根拠（E4）**: 公式推奨 — `@path` インポートでメインファイルを簡潔に保ちつつ、詳細は別ファイルで提供。

### Phase 5: 「なぜ」チェック（追加品質）

CLAUDE.md 内のルール・規約の品質を確認:

| # | チェック項目 | 合格基準 |
|---|---|---|
| Q1 | ルールに理由が付いている | 主要ルールの50%以上に理由（括弧書き、「理由:」、「〜のため」等）がある |
| Q2 | 権限境界が明確 | 「自由にやってよい / Always」「確認が必要 / Ask first」「禁止 / Never」の3段階が区別されている |

> **根拠（Q2）**: GitHub 2,500+リポジトリ分析で、3段階境界を持つ CLAUDE.md は AI 出力品質が有意に高い。

## Phase 6: スコア算出とレポート

### スコア計算

- 必須項目（D1, D2, D3, D4, V1）: 各2点（合計10点）
- 推奨項目（D5, D6, S1-S5, V2-V5, E1-E4）: 各1点（合計13点）
- 品質項目（Q1, Q2）: 各1点（合計2点）
- **満点: 25点**

### レベル判定

| スコア | レベル | 意味 |
|---|---|---|
| 21-25 | AI-Ready | AI エージェントが高品質な出力を安定して生成できる |
| 15-20 | Almost Ready | 基本は整っている。改善で大幅な品質向上が見込める |
| 9-14 | Needs Work | AI は使えるが、文脈不足で品質にムラがある |
| 0-8 | Not Ready | AI エージェントへの文脈提供が不十分 |

## 注意事項

- チェックは **読み取り専用**。ファイルの変更は行わない
- 各チェック項目は Grep / Glob / Read で機械的に判定する
- 判定が曖昧な場合は「⚠️ 要確認」として手動確認を促す
- `.claude/rules/` 内のルールファイルも補助的に参照する（CLAUDE.md に統合されていない規約がないか確認）
- D2 のバージョンチェックは正規表現 `\d+\.\d+` でスタック名近傍を検索する
