# スキルクリエーター

効果的なスキルを作成するためのガイド。

## スキルとは

スキルは Claude を汎用エージェントから専門エージェントに変換するモジュール式パッケージ。特定ドメインの「オンボーディングガイド」として、いかなるモデルも完全には持てない手続き的知識を提供する。

**提供するもの:**

1. **特化ワークフロー** — 複数ステップの手順（デプロイ、コードレビュー等）
2. **ツール統合** — 特定ファイル形式や API を扱う手順
3. **ドメイン専門知識** — 会社固有の知識・スキーマ・ビジネスロジック
4. **バンドルリソース** — スクリプト・参照資料・アセット

## コア原則

### 1. 簡潔さが鍵

コンテキストウィンドウは公共財。**Claude はすでに非常に賢い** — Claude が持っていない情報のみ追加する。冗長な説明より簡潔な例を優先。

### 2. 適切な自由度

| 自由度 | 使い所 | 例 |
|---|---|---|
| 高（テキスト指示） | 複数アプローチが有効、コンテキスト依存 | コンテンツ作成 |
| 中（擬似コード） | 望ましいパターンがある | API 呼び出しパターン |
| 低（具体スクリプト） | 繊細でエラーしやすい、一貫性が重要 | PDF 変換、DB マイグレーション |

### 3. 段階的開示

3段階でコンテキストを効率管理する:

1. **メタデータ**（name + description）— 常にコンテキスト内（~100語）
2. **SKILL.md 本文** — トリガー時のみ読み込み（500行以下）
3. **バンドルリソース** — Claude が必要と判断した時のみ

## スキルの構造

```
skill-name/
├── SKILL.md          (必須) frontmatter + 指示
└── バンドルリソース   (任意)
    ├── scripts/      実行可能コード（Python/Bash 等）
    ├── references/   必要時にコンテキストに読み込むドキュメント
    └── assets/       出力で使用するファイル（テンプレート、画像等）
```

### SKILL.md

- **Frontmatter（YAML）**: `name` と `description` が必須。description はスキルの主要トリガーメカニズム — 「いつ使うか」の情報はすべてここに書く（本文はトリガー後に読み込まれるため）
- **Body（Markdown）**: スキルの使用手順。命令形で記述

### バンドルリソースの使い分け

| ディレクトリ | 用途 | コンテキスト |
|---|---|---|
| `scripts/` | 毎回同じコードを書き直す処理を自動化 | 実行のみ（読み込み不要） |
| `references/` | ドメイン知識・スキーマ・API仕様 | 必要時に読み込み |
| `assets/` | テンプレート・画像・ボイラープレート | 出力で使用（読み込み不要） |

**重複を避ける**: 情報は SKILL.md か references/ のどちらか一方のみに置く。

### 含めないもの

README.md, CHANGELOG.md, INSTALLATION_GUIDE.md 等の補助ドキュメントは不要。AI エージェントが作業に必要な情報だけを含める。

## スキル作成プロセス（6ステップ）

### ステップ 1: 具体例で理解する

スキルの使用パターンが明確でない場合、ユーザーに質問して具体例を集める:

- 「どんな機能をサポートすべきか？」
- 「どのように使われるか例を挙げてもらえるか？」
- 「このスキルをトリガーすべきユーザーの発言は？」

1メッセージで質問しすぎない。最重要な質問から始め、必要に応じてフォローアップ。

### ステップ 2: 再利用リソースを計画する

各具体例を分析し、繰り返し必要になるリソースを特定する:

| 例 | 分析 | リソース |
|---|---|---|
| 「PDF を回転して」 | 毎回同じ回転コードを書く | `scripts/rotate_pdf.py` |
| 「Todo アプリを作って」 | 毎回同じボイラープレート | `assets/hello-world/` |
| 「ログインユーザー数は？」 | 毎回スキーマを再発見 | `references/schema.md` |

### ステップ 3: スキルを初期化する

新規作成時は `init_skill.py` でテンプレート生成:

```bash
scripts/init_skill.py <skill-name> --path <output-directory>
```

既存スキルの更新時はこのステップをスキップ。

### ステップ 4: スキルを編集する

**前提**: スキルは別の Claude インスタンスが使う。Claude にとって有益で自明でない情報を含める。

#### Frontmatter の書き方

```yaml
---
name: skill-name
description: スキルの機能と使用タイミングを三人称で記述。「いつ使うか」はここに全て書く。
---
```

- **description は常に三人称**: 「Excel を処理しレポートを生成する」（良い） / 「私が処理します」（NG）
- description に「いつ使うか」を全て含める。本文の「使用タイミング」セクションは Claude の助けにならない

> 高度な frontmatter フィールド（`argument-hint`, `disable-model-invocation`, `context`, `hooks` 等）や文字列置換（`$ARGUMENTS`, `${CLAUDE_SKILL_DIR}`）については [references/advanced-features.md](references/advanced-features.md) を参照。

#### Body の書き方

- 命令形で記述する
- 500行以下に保つ — 超えそうなら references/ に分離
- 複数バリエーション対応時はコアワークフローのみ SKILL.md に残し、詳細は references/ に

**段階的開示パターン:**

```markdown
## 高度な機能
- **フォーム入力**: [FORMS.md](references/forms.md) を参照
- **API リファレンス**: [REFERENCE.md](references/api.md) を参照
```

参照は SKILL.md から1階層のみ。100行超の参照ファイルには先頭に目次を付ける。

#### デザインパターン参照

- **複数ステップのプロセス**: [references/workflows.md](references/workflows.md)
- **出力フォーマット・品質基準**: [references/output-patterns.md](references/output-patterns.md)

### ステップ 5: パッケージ化する

```bash
scripts/package_skill.py <path/to/skill-folder> [output-dir]
```

自動検証（frontmatter、命名規則、description 品質）→ 通過後に `.skill` ファイル（zip形式）を生成。

### ステップ 6: 反復する

1. 実際のタスクでスキルを使用
2. 苦労や非効率に気づく
3. SKILL.md またはリソースの更新箇所を特定
4. 変更を実装して再テスト

**評価駆動開発（推奨）**: スキルなしで代表タスクを実行 → 失敗・不足を記録 → 3つ以上のテストシナリオを作成 → ギャップを埋める最小限の指示を書く → 比較改善。

**品質フィルタ**: 各指示に「エージェントはこの指示なしで間違うか？」と問う。Yes なら含める、No なら削除する。

**Plan-Validate-Execute パターン**: バッチ操作では「中間計画を生成 → 計画を検証 → 実行」の3段階で安全性を確保する。

## アンチパターン

| 避けること | 正しいアプローチ |
|---|---|
| Windows パス `scripts\helper.py` | Unix パス `scripts/helper.py` |
| 複数ライブラリを並列列挙 | デフォルト1つ + 代替はエスケープハッチ |
| 時間依存の記述 | 「現在の方法」と「旧パターン（deprecated）」に分ける |
| 同じ概念に異なる用語 | 1スキル内で用語を統一 |
| 参照の深いネスト | SKILL.md から1階層のみ |
