# review-lp-uiux — 包括 LP + UIUX ガイドラインレビュー

リリース前の包括的チェック。**全ページ・全コンポーネントを対象に**、全21項目を検査し詳細レポートを出力する。

## 実行戦略

### 原則: 全画面・全コンポーネントが対象

LP だけでなく、**プロジェクト内の全ページ・全共有コンポーネント**を検査対象とする。

### Step 1: 全対象ファイルの列挙

以下を **すべて** Glob で列挙し、検査対象リストを作成する:

```
src/pages/**/*.{tsx,jsx}          — 全ページ（1ファイルも漏らさない）
src/components/**/*.{tsx,jsx}     — 全コンポーネント
src/i18n/**/*.json                — 全翻訳ファイル
**/index.css, **/globals.css      — グローバルCSS
tailwind.config.*                 — Tailwind設定
package.json                      — 依存関係
**/index.html                     — メタタグ・OGP
public/**                         — 静的アセット
```

列挙したファイル数をレポート冒頭に記載する（例: 「検査対象: 42ファイル」）。

### Step 2: 並列サブエージェントによる多角的チェック

**時間がかかっても網羅性を優先する。** 以下のグループに分けてサブエージェントを並列起動:

| エージェント | 担当項目 | 検査対象 |
|-------------|---------|---------|
| Agent A: ビジュアル品質 | #0 brand, #3 spacing, #4 component, #5 animation | 全ページ・全コンポーネント |
| Agent B: カラー・タイポ | #1 color, #2 typo | 全ページ・全コンポーネント + CSS + Tailwind config |
| Agent C: レスポンシブ・a11y | #6 responsive, #11 a11y | 全ページ・全コンポーネント |
| Agent D: 多言語・画像 | #9 i18n, #10 image | 全翻訳ファイル + 全ページ + public/ |
| Agent E: LP 専用 | #7 structure, #8 pricing | LandingPage + i18n |
| Agent F: リリースチェック | #12 launch | 横断（メタ・法的・技術・多言語） |
| Agent G: インタラクション品質 | #13 loading, #14 error-state, #15 toast, #16 z-index | 全ページ・全コンポーネント |
| Agent H: コンテンツ・ナビ | #17 text-overflow, #18 navigation, #19 modal, #20 table | 全ページ・全コンポーネント |

サブコマンド指定時は該当エージェントのみ起動する。

### Step 3: 各ページの個別レポート

全ページについて、**ページ単位で**違反を整理する。「どのページに何の問題があるか」を明確にする。

## UIUX チェック項目（18項目）

### #0 brand — ブランド哲学

**コンセプト**: 「業界1/100の価格」× プレミアムUI。安っぽさが敵。

**全画面チェック**:
- 各ページを Read し、全体の印象を評価
- shadow, gradient, backdrop-blur 等のプレミアム要素の使用状況を Grep
- チープな要素（原色ベタ塗り、小さすぎる余白、統一感のないフォント）がないか
- 価格の安さを「高級感を保ったまま」伝えているか
- ダークモード/ライトモードで品質が落ちていないか

### #1 color — カラー

**ルール**:
- 各サービスが固有のブランドカラーを持つ（OKLCH Hue 差し替えでスケール生成）
- CSS変数 or Tailwind config でブランドカラーが定義されていること
- ハードコードされた hex/rgb 値が散在していないこと

**全画面チェック**:
1. `tailwind.config.*` を Read → ブランドカラーの定義を確認
2. `index.css` / `globals.css` を Read → CSS変数でのカラー定義を確認
3. **全コンポーネント・全ページ**で `#[0-9a-fA-F]{3,8}` や `rgb(` を Grep → ハードコード色を全件検出
4. oklch の使用状況を Grep
5. ブランドカラーが CTA・ヘッダー・カード等の主要要素に適用されているか、全ページで確認
6. SVG 内の外部ブランドカラー（Ethereum 等）は許容するが明記する

### #2 typo — タイポグラフィ

**ルール**:
- フォント: Inter + Noto Sans JP
- H1/H2 は text-xl 以上で最大2行に収まる長さ
- `word-break: auto-phrase` で日本語の自然な改行
- font-family 指定順序: Inter, "Noto Sans JP", sans-serif

**全画面チェック**:
1. CSS で font-family 定義を確認
2. **全ページ**の `<h1>`, `<h2>` のテキストサイズクラスを Grep
3. `word-break` の指定状況を全ファイルで Grep
4. フォント読み込み（@import / `<link>`）を確認
5. 各ページで見出しサイズの一貫性を確認

### #3 spacing — スペーシング

**ルール（LP向け）**:
- コンテナ幅: max-w-7xl (1280px)
- セクション間: py-20 〜 py-28
- カード内パディング: p-6 〜 p-7
- 余白は「贅沢」を演出する重要要素

**ルール（アプリ画面向け）**:
- `container` / `max-w-` でコンテンツ幅が制御されていること
- `p-0` や `m-0` でパディングが完全に消えていないこと
- LP 向けの `py-20` / `max-w-7xl` 基準はアプリ画面には適用しない

**全画面チェック**:
1. **LP**: `max-w-` の使用状況を Grep → 7xl 以外のコンテナ幅を検出。セクション要素の `py-` を確認
2. **アプリ画面**: コンテンツ幅の制御があるか、パディングが消えていないかを確認
3. カードコンポーネントの `p-` クラスを確認
4. ページ間でスペーシングルールが一貫しているか比較

### #4 component — コンポーネント

**ルール**:
- Primary CTA: ブランドカラー + glow（shadow-lg + shadow-[brand]/25 等）
- Secondary CTA: アウトライン（border + transparent背景）
- Ghost ボタン: 背景なし
- カード: border + rounded-xl or rounded-2xl
- 角丸: rounded-lg 以上（rounded-md 以下は非推奨）

**全画面チェック**:
1. ボタンコンポーネント（共通 + 各ページ）を全件特定し Read
2. CTA ボタンに glow があるか、全ページで確認
3. **全ファイル**で `rounded-sm`, `rounded-md` の使用箇所を検出
4. カードの rounded クラスを全件 Grep
5. ボタンスタイルの一貫性を全ページで比較

### #5 animation — アニメーション

**ルール**:
- Framer Motion を使用
- 控えめフェードイン: duration 0.4〜0.7s
- スクロールトリガー: whileInView 等
- 派手な演出禁止: bounce, rotate, scale > 1.05
- prefers-reduced-motion 対応推奨

**全画面チェック**:
1. `package.json` で `framer-motion` の依存を確認
2. **全ファイル**で `motion.` の使用状況を Grep
3. `duration:` の値を全件 Grep → 0.4〜0.7 の範囲外を全件検出
4. `whileInView` の使用を確認
5. `bounce`, `rotate`, `scale` の使用を全件検出
6. `prefers-reduced-motion` の対応状況を確認

### #6 responsive — レスポンシブ

**ルール**:
- モバイルファースト設計
- 5ブレークポイント: base / sm(640) / md(768) / lg(1024) / xl(1280)
- グリッドが適切にレスポンシブ対応

**全画面チェック**:
1. Tailwind config でブレークポイントのカスタマイズを確認
2. **全ページ**で `sm:`, `md:`, `lg:`, `xl:` プレフィックスの使用状況を確認
3. レスポンシブプレフィックスが一切ないページを検出
4. `grid-cols-` がレスポンシブ対応しているか全件確認
5. テキストサイズにレスポンシブプレフィックスがあるか全ページで確認
6. `hidden` / `block` と `sm:block` / `md:hidden` 等の表示切替パターンを確認
7. `flex-col` → `md:flex-row` 等、モバイル縦並び・デスクトップ横並びのパターンを確認
8. `w-full` / `w-[固定値]` がレスポンシブ切替なしに混在していないか
9. 見出し (`text-2xl` 等) に `md:text-4xl` 等のレスポンシブサイズ指定があるか

### #9 i18n — 多言語（ハードコード禁止は必須）

**ルール**:
- 対応言語: ja / en / th / zh-CN の4言語
- **UIに表示される日本語テキストの直書きは禁止** — `t()` 経由必須。違反は ❌
- i18n キーが4言語すべてに存在すること

**全画面チェック**:
1. 各言語ファイルのキー数を比較 → 差分があれば**欠損キーを全件リスト化**
2. **全ページ・全コンポーネント**で日本語文字を含む文字列リテラルを Grep → JSXテキストノード、placeholder/label/title/alt属性、toast メッセージに日本語リテラルがあれば ❌
3. **許容（偽陽性として除外）**: コメント、console.log/warn/error、import文、i18n非対応と明記されたページ、テストファイル
4. `t("` や `useTranslation` の使用状況を確認
5. 翻訳キーの命名規則の一貫性を確認

### #10 image — 画像

**ルール**:
- 画像の alt テキストが設定されていること
- 適切なフォーマット・サイズであること

**全画面チェック**:
1. **全ファイル**の `<img` タグの alt 属性を Grep → 空 alt を全件検出
2. `public/` 配下の画像ファイルを Glob で列挙 → フォーマット・サイズ確認
3. 1MB 超の画像がないか確認
4. next/image や lazy loading の使用状況を確認

### #11 a11y — アクセシビリティ

**ルール**:
- フォーカスリング: `focus-visible:ring-2` 等
- disabled 状態: `opacity-50 cursor-not-allowed` 等
- 見出し階層: h1 → h2 → h3 の順序（スキップなし）
- cursor-pointer: クリッカブル要素に適用
- aria-label: アイコンボタン等に設定

**全画面チェック**:
1. **全ファイル**で `focus-visible` or `focus:ring` の使用を Grep
2. `disabled` props とそのスタイリングを全件確認
3. **各ページ**の見出し階層を個別に確認（h1 → h3 のスキップ等）
4. `<button`, `<a` 要素に `cursor-pointer` があるか全件確認
5. アイコンのみのボタンに `aria-label` があるか全件確認
6. `role` 属性の適切な使用を確認

### #13 loading — ローディング状態

**ルール**:
- 非同期データ取得には必ずローディング表示を伴うこと
- Skeleton UI または Spinner を統一的に使用すること
- 楽観的 UI 更新が適切に使われていること

**全画面チェック**:
1. **全ページ**で `useQuery` / `useMutation` / `trpc.` の使用箇所を全件列挙
2. 各箇所の近傍に `isLoading` / `isPending` / `isFetching` の分岐があるか確認
3. ローディング表示が `Skeleton` / `Spinner` / `Loading` コンポーネントで統一されているか
4. ローディング中に操作不能になるボタンに `disabled` が設定されているか
5. `useMutation` で楽観的更新 (`onMutate` / `optimisticUpdate`) が適切に使われているか
6. ページ全体のローディング（初回描画）と部分ローディング（データ更新）が区別されているか

### #14 error-state — エラー・空状態（エラーUIは必須）

**ルール**:
- **`useQuery` を使う全コンポーネントに `isError` 分岐とエラー表示が必須** — 欠落は ❌
- リスト/テーブルのデータが0件の時は空状態 UI を表示すること
- エラーバウンダリで予期せぬクラッシュを捕捉すること

**全画面チェック**:
1. **全ページ**で `useQuery` / `trpc.*.useQuery` の使用箇所を全件列挙 → 同じコンポーネントスコープ内に `isError` / `error` の条件分岐があるか確認。**欠落は ❌**
2. エラー分岐はあるが `retry` / `refetch` ボタンがなければ ⚠️（推奨）
3. エラー表示が統一されたコンポーネントか、各ページでバラバラか
4. リスト/テーブル表示で `.length === 0` / `empty` / `no data` の空状態分岐があるか
5. 空状態に適切なイラスト/メッセージ/アクション（「最初の〜を作成」等）があるか
6. `ErrorBoundary` の設置状況を確認
7. ネットワークエラー（オフライン）時の表示を確認
8. 404 / 403 等のエラーページが存在するか

### #15 toast — トースト・通知

**ルール**:
- トースト通知は `sonner` 経由で統一すること
- `alert()` / `window.confirm()` / `window.prompt()` は使用禁止
- トーストの種類（success / error / info / warning）が適切に使い分けられていること

**全画面チェック**:
1. `package.json` で `sonner` の依存を確認
2. **全ファイル**で `toast(` / `toast.success` / `toast.error` の使用を Grep
3. `alert(` / `window.confirm(` / `window.prompt(` の使用を全件検出 → ❌
4. `useMutation` の `onSuccess` / `onError` でトーストが表示されているか
5. トーストの duration が統一されているか
6. 破壊的操作（削除等）に確認ダイアログ（`AlertDialog` 等のUIコンポーネント）が使われているか

### #16 z-index — z-index 管理

**ルール**:
- z-index はTailwind定義済みクラス (`z-10`, `z-20`, `z-50`) を使用すること
- 任意値 `z-[数値]` のハードコードは避けること
- レイヤー階層が設計されていること

**全画面チェック**:
1. **全ファイル**で `z-` クラスの使用を全件 Grep
2. `z-[数値]` の任意値を全件検出 → ⚠️
3. z-index の値が論理的な階層になっているか確認（モーダル > ドロップダウン > ヘッダー > コンテンツ）
4. `style={{ zIndex:` のインライン指定を全件検出 → ❌
5. 複数の固定要素（ヘッダー、サイドバー、フローティングボタン）間の重なり順を確認

### #17 text-overflow — テキスト切れ

**ルール**:
- 長文を表示する要素には切れ対策を施すこと
- 多言語対応のため、テキスト長が言語により大きく異なることを前提とすること

**全画面チェック**:
1. テーブルセル (`<td` / `<th`) に `truncate` / `max-w-` が設定されているか
2. カード内のタイトル/説明に `line-clamp-` / `truncate` があるか
3. ナビゲーション/タブのラベルが長文で折り返した時のレイアウト崩れリスク
4. `whitespace-nowrap` がレスポンシブ時に問題を起こさないか
5. i18n の各言語でテキスト長が極端に異なるキー（日本語 vs タイ語等）を特定
6. ツールチップやポップオーバーの内容が長文時に切れないか

### #18 navigation — ナビゲーション

**ルール**:
- ユーザーが現在地を把握できること
- 主要ページへのアクセスが2クリック以内であること
- モバイルナビゲーションが適切に実装されていること

**全画面チェック**:
1. サイドバー/ヘッダーにアクティブ状態の視覚的表示があるか（`active` / `aria-current` 等）
2. パンくずリスト（Breadcrumb）が深い階層で使用されているか
3. 「戻る」ボタンやナビゲーション手段がモーダル/詳細画面にあるか
4. モバイルハンバーガーメニューの開閉が実装されているか
5. メニューの開閉時にbody スクロールがロックされるか
6. ルーティング設定と実際のナビゲーションリンクの整合性

### #19 modal — モーダル・ダイアログ

**ルール**:
- モーダルはアクセシブルであること
- Esc キーで閉じられること
- 背景クリックで閉じられること（破壊的操作を除く）

**全画面チェック**:
1. `Dialog` / `Modal` / `Sheet` コンポーネントの使用箇所を全件列挙
2. 各モーダルに閉じるボタンが存在するか
3. `onOpenChange` / `onClose` ハンドラが設定されているか
4. フォーカストラップ（モーダル内にフォーカスが閉じ込められる）が実装されているか
5. 背景スクロール制御（`overflow-hidden` on body）があるか
6. モーダル内フォームの送信後に自動で閉じる/リセットされるか
7. ネストされたモーダル（モーダル内モーダル）がないか → ⚠️ アンチパターン

### #20 table — テーブル・リスト

**ルール**:
- データテーブルは操作しやすいこと
- 空状態が適切に表示されること
- モバイルで横スクロールまたは代替レイアウトがあること

**全画面チェック**:
1. `<table` / `Table` コンポーネントの使用箇所を全件列挙
2. ソート機能が必要なカラムに `sortable` / `onClick` があるか
3. ページネーション / 無限スクロールの実装があるか（大量データ対応）
4. モバイルでの表示方法（横スクロール `overflow-x-auto` / カードレイアウト切替 / レスポンシブ非表示カラム）
5. 行クリックやアクションボタンに `cursor-pointer` があるか
6. ヘッダー行が固定（`sticky`）されているか（長いテーブル）
7. データ0件時の空状態UIがあるか（#14 error-state と連携）

## LP チェック項目（3項目）

### #7 structure — LP構成（11セクション心理フロー）

| 順序 | セクション | 心理的役割 |
|------|-----------|-----------|
| 1 | Hero | 価格ショックで注目 |
| 2 | ソーシャルプルーフ | 信頼の土台 |
| 3 | 問題提起 | 課題の共感 |
| 4 | 解決策 | プロダクトが解決策 |
| 5 | 機能紹介 | 具体的な機能 |
| 6 | コスト訴求 | 価格ショック→理由→品質証明 |
| 7 | 品質証明 | 技術力・セキュリティ |
| 8 | 比較表 | 競合との比較 |
| 9 | FAQ | 疑問解消 |
| 10 | 最終CTA | 行動促進 |

**ユーザーガイド検査方法**:
1. `**/guide*`, `**/help*`, `**/onboarding*`, `**/tutorial*`, `**/how-to*` を Glob で探索
2. ルーティング設定で `/guide`, `/help`, `/docs` 等のパスがあるか確認
3. i18n キーに `guide`, `help`, `howTo`, `tutorial` 関連キーがあるか Grep
4. LP の FAQ やフッターからガイドへのリンクがあるか確認
5. 上記いずれもなければ ❌

**法的ページ検査方法**:
1. ルーティング設定で `/terms`, `/privacy`, `/legal`, `/tokushoho`, `/cookie-policy` 等のパスがあるか確認
2. `**/terms*`, `**/privacy*`, `**/legal*`, `**/tokushoho*` を Glob で探索
3. i18n キーに `terms`, `privacy`, `legal`, `tokushoho`, `cookie` 関連キーがあるか Grep
4. LP フッターから利用規約・プライバシーポリシーへのリンクがあるか確認
5. 特定商取引法に基づく表記に必要な項目（販売業者名、所在地、連絡先、価格、支払方法、引渡時期、返品規定）が含まれているか確認
6. 上記いずれかが欠落していれば ❌

**Cookie同意検査方法**:
1. `cookie`, `consent`, `CookieBanner`, `CookieConsent`, `gdpr` を全ファイルで Grep
2. Cookie同意バナー/モーダルのコンポーネントが存在するか確認
3. 同意前に分析系Cookie（Google Analytics等）やサードパーティCookieが発火していないか確認
4. 同意の保存（localStorage / Cookie）と再表示制御のロジックがあるか確認
5. プライバシーポリシーへのリンクがバナー内に含まれているか確認
6. Cookie を一切使用していない場合は「該当なし」として ✅ 扱い。ただし `document.cookie`, `localStorage`, `ga(`, `gtag(`, `analytics` を Grep して本当に不使用か確認する
7. Cookie/トラッキングを使用しているのに同意バナーがなければ ❌

**アプリ内ブラウザ対策検査方法**:
1. OAuth 認証を使用するページ（LP のログインボタン等）を特定
2. `inAppBrowser`, `in-app`, `openExternalBrowser`, `intent://`, `detectInApp` を全ファイルで Grep
3. アプリ内ブラウザ検出ロジックが存在するか確認（LINE / Facebook / Twitter(X) / Instagram / WeChat / 汎用 WebView）
4. 検出時のリダイレクト処理を確認:
   - LINE: `openExternalBrowser=1` パラメータ付加
   - Android: `intent://` URL でデフォルトブラウザ遷移
   - iOS: バナー案内（Safari / Chrome で開くよう誘導）
5. フォールバック UI（バナー/案内表示）が存在するか確認
6. OAuth 認証を使用しないアプリの場合は「該当なし」として ✅ 扱い
7. OAuth を使用しているのにアプリ内ブラウザ対策がなければ ❌

**領収書検査方法**:
1. 決済機能の有無を確認: `stripe`, `crypto`, `payment`, `billing`, `checkout` を Grep
2. 決済機能がある場合、`receipt`, `invoice`, `領収`, `明細` を全ファイルで Grep
3. Billing ページ等に決済履歴・領収書ダウンロード・明細表示があるか確認
4. Stripe の場合: Stripe Dashboard からの領収書リンク or 自前の領収書生成を確認
5. 決済機能があるのに領収書機能がなければ ❌

### #14 設計原則（全チェックの判断基準）
1. **安さは武器** — 価格の安さは堂々と見せるが、安っぽさは排除する
2. **1サービス1色** — ブランドカラーの重複を避け、サービスの識別性を確保する
3. **余白は贅沢** — 余白を惜しまない。詰め込みはチープさの原因
4. **先回り回答** — ユーザーの疑問を FAQ やセクション構成で先回りして解消する
5. **数字で語る** — 抽象的な表現より具体的な数字で価値を示す
6. **バリアゼロ** — 導入障壁を最小化する UI/UX
7. **多言語ネイティブ** — 翻訳ではなく、各言語でネイティブに読める品質
