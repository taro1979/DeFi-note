# レビュー視点の詳細

各サブエージェントに渡すプロンプトの要点。

## 1. バグ検出

重大なロジックバグに集中。以下を重点的にチェック:
- null/undefined アクセス（オプショナルチェーン漏れ）
- off-by-one エラー（配列インデックス、ループ境界）
- 条件分岐の論理誤り（&&/|| の混同、否定の漏れ）
- async/await の欠落（Promise が未解決のまま使用）
- 型アサーション（as）による型安全性の喪失
- エラーハンドリングの欠落（catch で握り潰し）

ニッチなエッジケースではなく、通常フローで踏むバグを優先。

## 2. セキュリティ

OWASP Top 10 を軸に、このプロジェクト固有のリスクも考慮:
- SQLインジェクション（Drizzle ORM の raw query 使用時）
- XSS（dangerouslySetInnerHTML、ユーザー入力の未エスケープ）
- 認証・認可の欠落（tRPC ルーターの protectedProcedure 漏れ）
- CSRF（状態変更エンドポイントの保護）
- 秘密鍵・APIキーのハードコード
- ログへの機密情報出力
- Postback API のなりすまし（api_key 検証の不備）
- click_id の推測可能性

## 3. アーキテクチャ準拠

プロジェクトの .claude/rules/ と CLAUDE.md に照らして確認:
- レイヤー依存方向: client→server の直接 import 禁止（tRPC経由のみ）
- shared/ は下位レイヤーのみ参照
- server/ 内で console.log 禁止（createLogger 使用）
- 新規ファイルが正しいレイヤーに配置されているか
- 環境変数は .env から読み込み
- 空実装（TODO のみ）禁止

## 4. パフォーマンス

実際にユーザー体感に影響する問題を検出:
- N+1 クエリ（ループ内の DB アクセス）
- 不要な再レンダリング（useEffect の依存配列、メモ化漏れ）
- 巨大データの一括ロード（ページネーション未対応）
- 不要な await の直列化（Promise.all で並列化可能）
- インデックスなしの頻出クエリパターン

マイクロ最適化は無視。測定可能な影響がある問題のみ。

## 5. データ整合性

DB スキーマ・Zod バリデーション・フロントエンド型の三層整合性:
- drizzle/schema.ts のカラムと Zod スキーマの不一致
- マイグレーション漏れ（スキーマ変更があるがマイグレーションなし）
- shared/ の型定義と実際の API レスポンスの乖離
- NOT NULL カラムに null を挿入する可能性
- enum 値の不一致（DB定義とコード定義）

## 6. git 履歴コンテキスト

変更ファイルの git blame と過去のコミットを確認:
- 過去に修正されたバグパターンの再導入
- revert されたアプローチの再実装
- 過去のPRコメントで指摘された問題の再発
- 意図的に残されたコメント（HACK, FIXME）の無視

`git log --oneline -20 -- <file>` と `git blame <file>` を使用。

## 7. エッジケース・境界値

通常テストで見落としやすい境界条件:
- 空配列・空オブジェクト・空文字列の処理
- 0, -1, Number.MAX_SAFE_INTEGER 等の境界値
- 並行リクエストによる競合状態（楽観ロック未使用）
- タイムゾーン・ロケール依存の日時処理
- Unicode 文字列（絵文字、結合文字）の処理
- ネットワークタイムアウト・部分的な障害
