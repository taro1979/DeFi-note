# Multi-Perspective Code Review

7つの独立した視点で並列レビューし、信頼度スコアで偽陽性を排除する。

## ワークフロー

### 1. 対象の特定

**PR番号/URLが指定された場合** → `gh pr diff` で差分取得
**ローカル差分の場合** → `git diff HEAD` または `git diff main...HEAD` で差分取得

差分が空なら終了。

### 2. コンテキスト収集（並列）

以下を並列で収集:
- 変更ファイル一覧と差分
- 関連する CLAUDE.md / .claude/rules/*.md の内容
- 変更ファイルの全文（差分の前後を理解するため）

### 3. 多角的レビュー（7つの並列サブエージェント）

**全エージェント共通**: `--model sonnet` で起動。各エージェントに差分・変更ファイル・関連ルールを渡す。

各エージェントは発見した問題を以下の形式で返す:
```
- ファイル: <path>:<line>
  問題: <説明>
  視点: <どの視点で発見したか>
  根拠: <なぜ問題と判断したか>
```

#### 視点一覧

詳細は [references/perspectives.md](references/perspectives.md) を参照。

| # | 視点 | 焦点 |
|---|------|------|
| 1 | バグ検出 | ロジックエラー、off-by-one、null参照、型不整合 |
| 2 | セキュリティ | インジェクション、認証漏れ、秘密鍵露出、OWASP Top 10 |
| 3 | アーキテクチャ準拠 | レイヤー依存方向、CLAUDE.md/rulesへの準拠 |
| 4 | パフォーマンス | N+1クエリ、不要な再レンダリング、メモリリーク |
| 5 | データ整合性 | スキーマとZodの不一致、マイグレーション漏れ、型の乖離 |
| 6 | git履歴コンテキスト | 過去の変更意図との矛盾、revert済みパターンの再導入 |
| 7 | エッジケース・境界値 | 空配列、空文字、上限値、並行処理の競合 |

### 4. 信頼度スコアリング（並列）

各発見に対し、独立した sonnet サブエージェントで信頼度を0-100で採点:

| スコア | 意味 |
|--------|------|
| 0-24 | 偽陽性。既存の問題、または精査に耐えない |
| 25-49 | 実際の問題かもしれないが検証不十分。スタイル指摘で rules に明記なし |
| 50-74 | 実際の問題だがニッチ。PR全体の重要度からすると低い |
| 75-89 | 高確信。実運用で発生しうる問題。既存アプローチが不十分 |
| 90-100 | 確定的。エビデンスで裏付けあり。頻繁に発生する |

### 5. フィルタリングと出力

**スコア75以上のみ**を最終レポートに含める。

#### 偽陽性の判定基準

以下は偽陽性として除外:
- 変更行以外の既存問題
- リンター・型チェッカー・CIが検出する問題
- 意図的な機能変更
- 一般的なコード品質（テストカバレッジ不足等）、rules に明記がない限り
- lint ignore コメントで明示的に抑制された問題

#### 出力フォーマット

```markdown
## Code Review

{N}件の問題を検出:

### 1. [視点] 問題の簡潔な説明
**ファイル**: `path/to/file.ts:42`
**信頼度**: 85/100
**詳細**: 問題の具体的な説明と根拠
**提案**: 修正案（あれば）

### 2. ...
```

問題が0件の場合:
```markdown
## Code Review

問題なし。バグ・セキュリティ・アーキテクチャ準拠・パフォーマンス・データ整合性・エッジケースを確認済み。
```

### 6. PR投稿（オプション）

PRレビューの場合、ユーザーの承認を得てから `gh pr comment` で結果を投稿。
