# 自己修正ドキュメントルール

## トリガーと必須アクション

### 1. スキーマ変更時
`drizzle/schema.ts` にテーブルやカラムを追加・変更・削除した場合:
- `docs/domain/data-model.md` のエンティティ一覧とテーブル詳細を更新
- `CLAUDE.md` の「ドメインモデル」行を更新
- `docs/architecture.md` のテーブル数記載を更新

### 2. 新規ファイル・ディレクトリ追加時
新しいソースファイルやディレクトリを追加した場合:
- `docs/architecture.md` の該当レイヤーにパスが記載されているか確認
- 記載がなければ追加（主要ファイルのみ。一時ファイルやテストは不要）

### 3. API エンドポイント変更時
tRPC ルーターや Express エンドポイントを追加・変更した場合:
- `docs/architecture.md` の RBAC 表を確認・更新

### 4. package.json scripts 変更時
scripts を追加・変更・削除した場合:
- `CLAUDE.md` の「コマンド」セクションを更新

### 5. 機能仕様書（spec/）変更時
`spec/` 配下の仕様書を追加・変更・削除した場合:
- `Spec.md`（プロジェクト概要仕様）の該当セクションを確認・更新
  - 新機能追加 → Phase 2 リストへの追加 or 本体セクションへの反映
  - データモデル変更 → 「データモデル」セクションを更新
  - フロー変更 → 「S2Sトラッキングフロー」等の該当フローを更新
- `CLAUDE.md` の「読み方ガイド」に新しい仕様書が必要なら追記

### 6. エージェントが躓いた時
ドキュメントの記載が古い・不正確で作業に支障が出た場合:
- 該当ドキュメントをその場で修正
- `lessons.md` に教訓を追記（同じ問題の再発防止）

## 原則
- ドキュメント修正は実装変更と同一コミットに含める
- 「後で直す」は禁止。変更時に即座に修正する
- 構造テスト（`server/structural.test.ts`）がドキュメント乖離を検出した場合、テストを修正するのではなくドキュメントを修正する
